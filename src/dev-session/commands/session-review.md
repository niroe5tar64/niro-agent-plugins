# /session-review

セッション状態を整合チェックし、矛盾・漏れ・危険な前提を抽出する。

## 必読

- プラグインルート配下の `docs/conversation-rules.md`
- プラグインルート配下の `docs/state-files.md`

このコマンドはセッション全体を横断して矛盾を見つける必要があるため、推論密度が要る。
高精度モデルで呼び出すことが望ましい（呼び出し時のモデルを継承する）。

## 入力

- 引数: `session-name`（任意）
- 無ければ最新 mtime のセッションを対象にする。

## 手順

### Step 1: 9 ファイルを Read

セッションディレクトリの全 9 ファイルを読み込む。読まずに推論しない。

### Step 2: 観点ごとに精査

以下の観点で問題を抽出する。

| 観点 | チェック内容 |
|---|---|
| 矛盾 | `decisions.md` と `assumptions.md` の衝突、`decisions.md` 内の自己矛盾 |
| ゴール未充足 | `goal.md` のゴールに対して、`decisions.md` だけで達成できるか |
| 危険な前提 | `assumptions.md` のうち、誤っていると実装が破綻する前提 |
| 仕様漏れ | 既存仕様（`context.md`）から派生する未確認の論点が `open-questions.md` に出ているか |
| テスト観点不足 | `findings.md` / `action-items.md` にテスト観点が欠落していないか |
| スコープ漏れ | `decisions.md` の Non-goals 相当が明示されているか |
| 実装リスク | `current-state.md` の `mode` に応じたリスク（debug なら再現条件、design なら拡張性 等） |

### Step 3: 重大度判定と最大 5 件への絞り込み

- 重大度: `high` / `medium` / `low`
- `high` から優先し、最大 5 件まで絞る。
- 各件について「修正案 1 つ」を併記する（複数案を並べない）。

### Step 4: `findings.md` への追記

末尾に以下のセクションを追加する（`docs/state-files.md` の雛形参照）。

```md
## Review Results (<YYYY-MM-DD HH:mm>)

- [<重大度>] <該当ファイル> — <一行説明> / 提案: <修正案>
```

### Step 5: ユーザーへ報告

抽出した問題を簡潔なリスト形式で提示する。

- 各行: `[重大度] 該当ファイル — 説明 / 提案`
- 最大 5 行
- 末尾に「どれから対応しますか？」と 1 質問だけ添える

問題が 0 件の場合は「整合性に問題なし。次は何を進めますか？」と簡潔に返す。

## 出力

- 抽出された問題のリスト（最大 5 行）
- 次の 1 質問

返答全体で 3〜10 行に収める（review はやや長め可）。

## 禁止

- 状態ファイルの自動修正（提案のみで、実際の書き換えはユーザー承認後）
- ファイルの削除
- 6 件以上の指摘を一度に出す
- 6 行を超える説明を 1 件あたりに付ける
- 実装やコミットを行わない
